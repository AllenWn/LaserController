# NIR 激光控制器固件设计说明

## 1. 文档目的

本文档从系统逻辑层面对手持式近红外外科激光控制器的固件设计进行说明。

本文档的目标不是描述源代码结构或实现细节，而是定义系统的目标行为，并清晰到仅凭本文档就能够复现同样的固件架构与控制逻辑。

因此，本文档用于验证以下内容：

- 系统的控制哲学
- 系统的安全模型
- 状态机行为
- 输入信号、安全判定与输出行为之间的关系
- 系统在启动、正常运行和故障情况下的预期行为

如果后续代码与本文档不一致，应以本文档作为设计参考，并将代码修正为符合本文档的设计逻辑。

## 2. 系统使命

本固件运行在 `ESP32-S3-WROOM-1U` 上，作为整个激光设备的监督控制器。

MCU **不直接实现**以下功能：

- 激光电流闭环调节
- TEC 温度闭环调节
- USB-C Power Delivery 协议协商

这些功能由专用硬件完成。

MCU **负责实现**以下功能：

- 系统启动监督
- 安全状态监测
- 状态机控制
- 激光发射许可控制
- 故障分类与恢复策略
- LD、TEC 与 DAC 设定值门控的输出协调

因此，该固件的本质是一个实时监督安全控制器。

## 3. 顶层控制哲学

整个系统围绕一个核心问题展开：

**当前是否允许激光发射？**

为了回答这个问题，固件会综合考虑以下信息：

- 硬件电源良好信号
- 硬件环路良好 / 温度良好信号
- IMU 姿态安全状态
- 操作员触发输入
- 内部故障历史与锁存状态

这个判定结果将控制：

- 系统能否进入发射状态
- LD 输出控制线是否可以进入工作模式
- 系统是否能够保持在上电待命状态
- 系统是否必须切换进入故障状态

因此，固件具有两个彼此相关但不同的职责：

1. 让硬件安全地上电并持续受监督
2. 安全地允许或拒绝激光发射

这两个职责必须分开，因为“设备已上电”和“允许激光发射”不是同一个条件。

## 4. 安全模型

系统安全模型是分层设计的。

### 4.1 硬件级安全

最快、最本地化的保护由硬件实现：

- LD driver 自身完成激光电流调节
- TEC controller 自身完成温度调节
- 电源控制器输出 `PGOOD` 信号
- 驱动器 / 控制器硬件自身输出“good”或监测信号

固件不替代这些保护，而是对其进行监督。

### 4.2 固件级监督安全

固件持续检查整个系统是否仍处于允许的工作范围内。

这包括：

- TEC 电源域是否健康
- LD 电源域是否健康
- TEC 控制器是否报告热状态正常
- LD 驱动器是否报告环路状态正常
- 当前设备姿态是否允许使用
- 操作员当前是否请求发射

### 4.3 失效安全策略

只要存在不确定性，系统就优先选择更安全的动作。

例如：

- 如果安全数据不再是最新有效数据，则该条件视为不安全
- 如果某个必需的监测任务停止更新，则拒绝发射
- 如果发生内部电气故障，则输出退回到安全状态
- 如果操作行为不安全，即使系统仍然带电，也必须拒绝发射

## 5. 功能分层

固件在概念上分为五层。

### 5.1 硬件接口层

该层定义 MCU 所连接的信号和通信通道。

包括：

- 数字 GPIO 输入
- 数字 GPIO 输出
- 与 IMU 的 SPI 通信
- 与 DAC 的 I2C 通信
- 已规划但尚未完全确定设计的 ADC 模拟采样链路

### 5.2 监测层

该层对各子系统进行采样，并将原始信号转换为子系统级别的健康结果。

例如：

- 电源监测器评估 `PWR_TEC_GOOD` 和 `PWR_LD_PGOOD`
- LD 监测器评估 `LD_LPGD`，以及后续的 LD 模拟遥测
- TEC 监测器评估 `TEC_TEMPGD`，以及后续的 TEC 模拟遥测
- IMU 监测器评估姿态安全
- Trigger 输入监测器评估操作者触发状态

该层回答的问题包括：

- LD driver 是否健康？
- TEC controller 是否健康？
- 当前姿态是否允许使用？
- 操作者是否正在按下触发？

### 5.3 系统状态层

该层存储最新的、经过验证的各子系统快照。

它提供统一的全局视图，包括：

- 电源状态
- LD 状态
- TEC 状态
- IMU 状态
- 故障锁存状态

该层的目的是将采样时序与监督决策时序解耦。

### 5.4 Supervisor 层

这是核心安全决策层。

它负责：

- 新鲜度检查
- 电源健康判断
- 发射许可判断
- 故障分类
- 故障锁存与清除策略
- DAC 钳位策略
- 状态机输入生成

该层是系统最终的决策者，用于决定：

- 系统是否允许继续保持上电
- 激光是否允许发射
- 系统是否必须进入故障行为

### 5.5 状态机与输出层

该层表达系统的全局状态，以及每种状态下对应的目标输出行为。

它控制：

- TEC 电源使能
- LD 电源使能
- 通过 `SBDN` 控制 LD 工作模式
- DAC 是否必须被钳位到安全值

该层将 supervisor 的安全决策转换为稳定、明确的系统行为。

## 6. MCU 接口总结

本节描述 MCU 关键接口的逻辑含义。

### 6.1 系统 / 调试 / 启动接口

- `EN`：ESP32 复位 / 使能输入，用于 MCU 复位，不属于运行期安全逻辑
- `GPIO0`：启动配置引脚，保留
- `GPIO46`：启动选项引脚，保留
- `UART0 TX/RX`：用于日志、调试与维护可视化
- `USB D+ / D-`：USB 通路，当前不属于激光安全闭环的一部分

### 6.2 电源控制接口

- `PWR_TEC_EN`：数字输出，控制 TEC 控制器电源域使能
- `PWR_TEC_GOOD`：数字输入，高电平表示 TEC 电源域正常
- `PWR_LD_EN`：数字输出，控制 LD driver 电源域使能
- `PWR_LD_PGOOD`：数字输入，高电平表示 LD driver 电源域正常

这些信号属于电源健康链路，而不仅仅是发射许可链路。

### 6.3 LD Driver 接口

- `LD_SBDN`：三电平控制信号，用于选择 LD 模式
- `LD_LPGD`：数字输入，高电平表示 LD 环路正常
- `LD_PCN`：数字输出，用于选择驱动器电流支路
- `LD_TMO`：来自 LD driver 的模拟监测输出
- `LD_LIO`：来自 LD driver 的模拟监测输出

对 `LD_SBDN` 的设计假设为：

- 低电平 = shutdown
- 高阻 = standby
- 高电平 = operate

对 `LD_PCN` 的设计假设为：

- 当前硬件版本不使用低电流支路
- 该信号固定为高支路选择

### 6.4 TEC Controller 接口

- `TEC_TEMPGD`：数字输入，高电平表示 TEC 热状态正常
- `TEC_TMO`：来自 TEC controller 的模拟监测输出
- `TEC_ITEC`：表示 TEC 电流的模拟监测输出
- `TEC_VTEC`：表示 TEC 电压的模拟监测输出

### 6.5 IMU 接口

- SPI 时钟、MOSI、MISO、CS
- IMU 可选中断输出

IMU 用于判断激光器是否处于允许的姿态范围内。

当前设计策略为：

- 激光设计为向下工作
- 如果光束方向上抬超过允许角度阈值，则必须禁止发射

### 6.6 Trigger 输入

- 预留一个外部 GPIO 作为 trigger 输入
- trigger 被视为数字型操作员请求信号
- trigger 本身绝不直接驱动激光
- 只有在系统已经处于允许的 ready 条件下，trigger 才具有效果

### 6.7 DAC 接口

- DAC 通过 I2C 连接
- 一个通道用于 LD 设定值控制
- 一个通道用于 TEC 设定值控制

DAC 不是主安全决策链路，而是从属于安全控制器：

- 安全状态下 -> DAC 输出钳位到安全值
- 发射状态下 -> 允许输出激活的设定值

### 6.8 预留 / 扩展接口

部分 GPIO 通过板对板连接器引出，用于后续集成：

- trigger
- 距离传感器
- 外部子板逻辑
- 潜在的 PD 相关辅助功能

在最终板级用法确认之前，这些接口不会纳入核心逻辑。

## 7. 核心状态

系统状态机在逻辑上定义为：

- `OFF`
- `POWERUP_TEC`
- `POWERUP_LD`
- `READY`
- `FIRING`
- `FAULT_INTERNAL`
- `FAULT_USAGE`

这些状态用于描述整个监督控制行为。

### 7.1 OFF

含义：

- 系统处于安全关闭状态
- 两路电源使能均为低
- 激光驱动器处于 shutdown
- DAC 输出被钳位到安全值

作用：

- 上电后的安全静止状态
- internal fault 清除后的恢复入口

### 7.2 POWERUP_TEC

含义：

- 正在开启 TEC 电源域
- LD 电源域保持关闭
- 激光发射不可能发生

退出条件：

- TEC 的 power-good 变为有效且稳定

失败条件：

- TEC 的 power-good 在超时时间内没有到达

### 7.3 POWERUP_LD

含义：

- TEC 电源域已经开启
- 现在开始开启 LD 电源域
- 激光发射仍然不允许

退出条件：

- LD 的 power-good 变为有效且稳定

失败条件：

- LD 的 power-good 在超时时间内没有到达

### 7.4 READY

含义：

- 两个控制器电源域都已经上电
- 所有必需安全条件都被持续监督
- 激光当前不发射
- LD 保持 standby，而不是 operation

在该状态下，设备已经做好发射准备，但尚未被命令发射。

### 7.5 FIRING

含义：

- 所有必需安全条件均成立
- 操作员 trigger 有效
- LD 被允许进入 operation 模式
- 有效 DAC 设定值被允许输出

这是唯一一个逻辑上允许激光发射的状态。

### 7.6 FAULT_INTERNAL

含义：

- 发生了内部电气、电源或控制路径故障
- 从控制器角度看，系统必须被认为是不可信的

输出策略：

- TEC 和 LD 电源使能都被关闭
- LD 被强制 shutdown
- DAC 输出被钳位

恢复策略：

- trigger 必须释放
- 必须满足清除条件
- 系统返回 `OFF`
- 然后重新执行完整上电序列

### 7.7 FAULT_USAGE

含义：

- 操作者行为或使用条件违反了允许的使用安全边界
- 系统硬件本身不一定损坏或失稳

示例：

- IMU 姿态不安全
- 未来的距离超范围条件

输出策略：

- 电源域保持开启
- LD 被强制 shutdown
- DAC 输出被钳位

恢复策略：

- trigger 必须释放
- 使用安全条件必须重新变为有效
- 系统返回 `READY`

## 8. 状态转移逻辑

### 8.1 启动路径

正常启动路径为：

`OFF -> POWERUP_TEC -> POWERUP_LD -> READY`

这意味着系统绝不会在复位后直接跳入一个可发射状态。

### 8.2 Ready 到 Firing

从 `READY` 进入 `FIRING` 必须同时满足：

- trigger 被按下
- emission permit 为真

只要任何一个条件不满足，系统就保持在 `READY`。

### 8.3 Firing 退出

在以下任一情况下，系统必须立即退出 `FIRING`：

- trigger 被释放
- permit 丢失
- 故障条件被锁存

### 8.4 故障优先级

Internal fault 优先级最高。

如果 usage 与 internal fault 同时存在：

- internal fault 行为优先生效
- 系统状态进入 `FAULT_INTERNAL`

这样可以保证电气 / 电源故障总是覆盖较弱的过程类故障。

## 9. 上电安全逻辑

上电不是一个单一动作，而是一条受监督的序列。

### 9.1 为什么需要上电序列

TEC controller 与 LD driver 是两个独立的硬件域。

固件必须确认：

- TEC 域存在且稳定
- LD 域存在且稳定
- 两个控制器都报告健康状态

只有在这些条件都满足后，系统才应该进入 ready 状态。

### 9.2 上电顺序

预期顺序是：

1. 从完全安全状态启动
2. 使能 TEC 电源域
3. 等待 TEC `PGOOD`
4. 使能 LD 电源域
5. 等待 LD `PGOOD`
6. 当两个电源域都正常后，再检查运行期安全条件
7. 进入 `READY`

### 9.3 超时保护

每个上电阶段都有各自的超时时间。

如果预期的 power-good 反馈未能按时到达：

- 该情况被视为 internal fault
- 系统切换进入 `FAULT_INTERNAL`

这可以防止设备长时间停留在“半上电”的不确定状态。

## 10. 发射许可逻辑

发射许可是决定当前是否允许激光工作的一组逻辑条件。

permit 的判断要求所有关键安全条件同时成立。

在当前设计阶段，permit 需要满足：

- TEC 电源域 ready
- LD 电源域 ready
- TEC 数字健康状态正常
- LD 数字健康状态正常
- IMU 姿态安全
- 没有 internal fault 被锁存
- 没有 usage fault 被锁存

操作员 trigger **不是** permit 本身的一部分。

而是：

- permit 回答的是“如果此刻收到请求，系统是否允许发射？”
- trigger 回答的是“操作员现在是否正在请求发射？”

只有二者同时成立，才能进入 firing。

## 11. 故障分类

故障被有意划分为两类。

### 11.1 Internal Fault

Internal fault 表示控制器级别或电气级别的不稳定。

当前 internal fault 来源包括：

- 运行中 `PGOOD` 丢失
- 上电超时
- 发射过程中 LD 或 TEC 数字故障

未来可能加入的 internal fault 来源包括：

- 模拟过温
- 模拟过流
- DAC 通信失败
- 监测任务失去更新

设计意图是：

- 任何 internal fault 都意味着当前供电控制路径不再可信
- 最安全的恢复方式是断电并重新执行受监督上电序列

### 11.2 Usage Fault

Usage fault 表示设备使用超出了允许的操作安全边界，但硬件本身未必损坏。

当前 usage fault 来源：

- 发射过程中 IMU 姿态不安全

未来可能加入的 usage fault 来源包括：

- 距离传感器超出范围
- 缺少组织接触条件
- 操作员过程类联锁失败

设计意图是：

- 必须立即停止发射
- 系统可以保持上电
- 在条件纠正后允许返回 `READY`

## 12. 故障检测与反应

### 12.1 Internal Fault 反应

当检测到 internal fault 时：

- 将故障锁存
- 状态切换到 `FAULT_INTERNAL`
- 两路电源使能关闭
- LD 强制 shutdown
- DAC 输出钳位到安全值

### 12.2 Usage Fault 反应

当检测到 usage fault 时：

- 将故障锁存
- 状态切换到 `FAULT_USAGE`
- 电源保持开启
- LD 强制 shutdown
- DAC 输出钳位到安全值

### 12.3 为什么 Usage Fault 也要有明确状态

Usage fault 使用明确的 fault 状态，而不是只用 permit 拉低来表示。

这样做的重要原因是：

- UI 可以区分“已 ready 但在等待”与“当前存在使用违规”
- 日志可以区分“操作违规”与“内部电气故障”
- 恢复策略可以与 internal fault 明确区分

## 13. 故障清除与恢复策略

### 13.1 操作者释放要求

两类 fault 都要求 trigger 被释放并稳定保持一段时间后，才允许恢复。

这可以防止操作者一直按住 trigger，使系统在故障一解除后立即再次进入 firing。

### 13.2 Usage Fault 清除

Usage fault 可以在以下条件满足时清除：

- usage 安全条件恢复
- trigger 已释放足够长时间以通过去抖逻辑
- 没有 internal fault 存在

之后系统返回 `READY`。

### 13.3 Internal Fault 清除

Internal fault 只能在以下条件满足时清除：

- 清除条件满足
- trigger 已释放足够长时间以通过去抖逻辑

之后系统返回 `OFF`，并重新执行：

`OFF -> POWERUP_TEC -> POWERUP_LD -> READY`

这一步是故意设计的，用于强制重新验证电源路径。

### 13.4 当前清除条件的范围

在当前设计阶段，清除条件主要绑定到恢复的操作安全条件，尤其是 IMU 安全。

后续应进一步细化，使每种 fault 来源具有各自独立的 clear 策略。

未来可改进的方向包括：

- internal 电源故障需要显式 service clear
- 热故障需要等待冷却时间
- 通信故障需要设备重新确认响应后才允许 clear

## 14. LD 输出控制逻辑

激光驱动器通过三种逻辑上彼此独立的方式受控。

### 14.1 电源使能

- `PWR_LD_EN` 控制 LD 控制器电源域是否上电
- `PWR_TEC_EN` 控制 TEC 控制器电源域是否上电

它们属于系统级安全与上电监督逻辑的一部分。

### 14.2 通过 SBDN 选择 LD 模式

LD driver 使用一个三电平控制输入。

设计含义为：

- `shutdown`：激光禁用
- `standby`：硬件已上电并准备就绪，但不发射
- `operate`：允许进入发射路径

逻辑使用方式：

- `OFF`、`POWERUP_TEC`、`POWERUP_LD`、`FAULT_INTERNAL`、`FAULT_USAGE` -> `shutdown`
- `READY` -> `standby`
- `FIRING` -> `operate`

### 14.3 PCN 支路选择

当前硬件版本只使用高电流支路。

因此：

- `PCN` 被视为固定配置输出
- 它不属于主动安全决策路径

## 15. DAC 控制逻辑

DAC 用于生成 LD 与 TEC 相关的模拟设定参考值。

DAC 从属于安全逻辑。

### 15.1 安全策略

在系统不允许发射时：

- DAC 输出钳位到安全值

### 15.2 Ready 策略

在 `READY` 状态下：

- TEC 通道可以维持待机 / ready 温度设定值
- LD 通道保持钳位或维持非发射安全参考值

### 15.3 Firing 策略

在 `FIRING` 状态下：

- 允许激活 LD 设定值
- TEC 的 ready 控制可以继续保持

确切的 DAC 原始数值属于硬件标定参数，而不是本逻辑设计文档要定义的内容。

## 16. 实时任务模型

固件采用 RTOS 下的周期性监督架构。

逻辑模型为：

- 快速数字监测任务周期运行
- IMU 采集任务周期运行
- 模拟采集任务周期运行
- supervisor 以最高控制优先级周期运行
- DAC 更新逻辑由 supervisor 的事件驱动

### 16.1 为什么使用这种架构

不同信号的时序需求不同。

- 数字状态线读取成本低，应该更频繁检查
- IMU 需要总线通信，但仍需相对较快更新
- 模拟遥测可以慢一些
- 最终安全决策应基于一致的系统快照

因此，该架构将采样与决策分离。

### 16.2 基于快照的监督

Supervisor 不应该混合使用来自不同时间点的半更新信息。

而应采用：

- monitor 任务发布最新子系统快照
- supervisor 读取最新的全系统快照
- supervisor 基于该快照做出一致决策

这让整个时序模型更容易被分析和验证。

## 17. 新鲜度规则

每个关键监测源都必须只在“最近刚更新过”的前提下才被视为有效。

如果某个监测源变得过期：

- 它就不再可信
- 所有依赖它的安全判断都应 fail safe

这样可以防止：

- 任务卡死
- 总线故障
- 传感器更新静默丢失

## 18. 已经确定的设计项与仍然开放的设计项

### 18.1 已确定的设计项

- 固件的整体职责
- 状态机结构
- 上电顺序设计
- trigger + permit 的 firing 规则
- internal 与 usage fault 分类
- 故障反应策略
- SBDN 三电平控制策略
- DAC 从属安全策略
- MCU 主要接口映射

### 18.2 仍开放的设计项

- trigger 最终 GPIO 与有效电平
- 距离传感器最终接口形式及其 fault 角色
- IMU 是否足以作为 usage fault clear 条件，还是还要加入 distance
- 模拟阈值最终定义
- 模拟电压到物理量的换算公式在板级标定后的最终形式
- 某些 internal fault 是否需要显式维护 clear，而不是 release 后自动 clear
- 数字关键 fault 是否应额外提供基于中断的紧急关断路径

## 19. 下一阶段设计扩展

在当前设计阶段之后，逻辑上预期还会加入以下内容：

- 将距离传感器逻辑并入 usage 安全条件
- 将 LD / TEC 的模拟监测结果并入 internal fault 分类
- 为外部设备加入更明确的通信失败处理
- 为关键数字 fault 加入可选的中断辅助快速关断
- 为不同 fault 来源提供更细粒度的 clear 策略

## 20. 设计总结

该固件被设计为一个面向激光系统的实时监督安全控制器，其下层 LD 与 TEC 控制都由独立硬件完成。

整个设计基于以下原则：

- 电源监督与发射许可是两个不同决策
- 只有在所有必要安全条件都成立时才允许激光发射
- 内部电气故障与操作过程故障不是同一类问题，必须区别处理
- 启动必须是受监督、分步验证的，而不能想当然
- 恢复必须依赖操作员释放和条件恢复
- 所有不确定性都应收敛到更安全的行为

由此得到的完整逻辑是：

- 安全启动
- 分阶段上电
- 验证子系统健康状态
- 在 ready 中保持激光待机
- 只有 permit 与 trigger 同时为真时才允许发射
- 根据故障含义进行分类
- internal 与 usage 违规采用不同反应策略
- 仅通过受控、已验证的状态转移进行恢复

本文档即为该固件系统的逻辑设计参考。
