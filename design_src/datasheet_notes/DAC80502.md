# DAC80502

Source PDF: `design_src/datasheet/dac80502.pdf`

Firmware-oriented notes for the **DAC80502** dual DAC used on the `MainBoard`.

## 1. Device role

- Dual-channel voltage-output DAC
- Supports either `SPI` or `IÂ²C`
- Current board revision uses it in `IÂ²C` mode

The DAC does not directly decide laser safety. It provides analog setpoints to hardware control loops.

## 2. Relevant pins

From the datasheet pin table:

- `VDD`
- `VOUTA`
- `RSTSEL`
- `AGND`
- `SPI2C`
- `SCLK/SCL`
- `SYNC/A0`
- `SDIN/SDA`
- `VOUTB`
- `VREFIO`

## 3. Interface selection

### `SPI2C`

Datasheet meaning:

- `0` -> SPI mode
- `1` -> IÂ²C mode
- Must remain static after power-up

Project implication:

- The mainboard straps `SPI2C` high
- Firmware must treat this part as an `IÂ²C` peripheral

## 4. Address selection

### `SYNC/A0`

In `IÂ²C` mode this pin becomes the address-select input.

Datasheet meaning:

- The device address starts with fixed upper bits `1001`
- The effective address depends on how `A0` is strapped
- `A0` can be tied to `VDD`, `AGND`, `SCL`, or `SDA`

Project implication:

- Current mainboard straps `A0` low
- The board notes identify the device at `0x48`

## 5. Power-up behavior

### `RSTSEL`

Datasheet meaning:

- `RSTSEL = AGND` -> power-up output is `zero-scale`
- `RSTSEL = VDD` -> power-up output is `mid-scale`

Project implication:

- The board is configured for safe zero-scale startup
- Firmware should still explicitly write startup codes so the analog command state is deterministic

## 6. Output channels in this project

From the mainboard schematic:

- `VOUTA -> DAC_OUTA` -> laser-driver analog command path
- `VOUTB -> DAC_OUTB` -> TEC-controller analog command path

Firmware implication:

- Channel A is laser-related
- Channel B is TEC-related
- Both channels should be clamped to known-safe codes during non-ready states

## 7. Reference behavior

### `VREFIO`

Datasheet meaning:

- Reference pin used either as internal-reference output or external-reference input

Project implication:

- This board uses the DAC in the usual `0 V` to `2.5 V` style control role
- Exact gain/reference configuration must match the final board stuffing and register setup

## 8. IÂ²C transaction model

Datasheet highlights:

- Device uses a standard slave address phase, command byte, and data bytes
- Several control registers are available, including config, gain, and channel data registers

Firmware implication:

- A real driver is required
- Writing DAC code is not just setting a GPIO level
- Startup and state transitions should explicitly push known DAC values

## 9. Project-specific interpretation

The DAC should be treated as:

- a real communication peripheral
- safety-adjacent, because it influences current and temperature command paths
- but not part of the primary fast interlock path
