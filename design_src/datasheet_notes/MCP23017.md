# MCP23017

Source PDF: `design_src/datasheet/MCP23017.pdf`

Firmware-relevant summary for the **MCP23017** I²C GPIO expander used on the `Auxiliary board`.

## 1. What it is

- 16-bit GPIO expander with **I²C** interface
- Two 8-bit ports:
  - `GPIOA` = `GPA0..GPA7`
  - `GPIOB` = `GPB0..GPB7`
- Supports **interrupt-on-change**
- `INTA` and `INTB` can operate independently

## 2. Operating limits relevant to firmware

- Supply range: approximately **1.8 V to 5.5 V**
- I²C clock rates supported by datasheet family:
  - `100 kHz`
  - `400 kHz`
  - `1.7 MHz`

For this project, it is powered from `VDD_3V3`, so the device is in a normal 3.3 V logic domain.

## 3. Pin groups relevant to this project

From the datasheet pinout and the `Auxiliary board` schematic:

- `VDD`: supply
- `VSS`: ground
- `SCL`, `SDA`: I²C bus
- `A0`, `A1`, `A2`: I²C address strap inputs
- `RESET`: hardware reset input
- `INTA`, `INTB`: interrupt outputs
- `GPA0..GPA7`, `GPB0..GPB7`: programmable GPIO

## 4. Pins used in this design

Based on the auxiliary schematic:

- `SDA` -> shared auxiliary I²C bus
- `SCL` -> shared auxiliary I²C bus
- `A0`, `A1`, `A2` -> tied low
- `RESET` -> tied high to `VDD_3V3`
- `INTA` -> exported as `GPIO_INT_A`
- `INTB` -> unused
- `GPA0..GPA3` -> button inputs
- `GPB4` -> `LRA_EN`

## 5. I²C address meaning

The device address is selected by `A2:A1:A0`.

In this design:

- `A0 = 0`
- `A1 = 0`
- `A2 = 0`

So the chip is strapped to its **base I²C address** configuration.

Practical firmware implication:

- The expander address is fixed by hardware in this design
- Firmware should not treat it as runtime-configurable

## 6. Interrupt behavior

Important firmware point:

- `INTA` / `INTB` are configurable interrupt outputs
- The datasheet supports:
  - interrupt-on-change
  - polarity control
  - mirrored or separate interrupt behavior

In this project:

- only `INTA` is routed back to the host side as `GPIO_INT_A`

That means firmware can use two strategies:

1. pure polling over I²C
2. interrupt-assisted expander servicing using `GPIO_INT_A`

For the final system, the second approach is preferable if the cross-board mapping confirms the main MCU really receives that interrupt line.

## 7. GPIO electrical meaning

Each `GPAx/GPBx` pin can be configured as:

- input
- output

If used as input, the logic level is determined by the external circuit.

In this board:

- the button lines are pulled up to `VDD_3V3`
- pressing the button shorts the signal to `GND`

So the button signals connected through the MCP23017 are **active low**.

## 8. What firmware should care about

For this project, the MCP23017 matters because it likely sits between the MCU and the user buttons.

Key firmware takeaways:

- Trigger / user button inputs may **not** be direct ESP32 GPIO inputs
- They likely have to be read from MCP23017 registers over I²C
- `GPIO_INT_A` can be used as an event line to reduce polling latency
- `LRA_EN` is generated by the expander, so haptic enable may also be indirect

## 9. Design-specific conclusions

- This device is **part of the control interface**, not the laser safety power path
- It is important for:
  - trigger input
  - auxiliary buttons
  - haptic control gating
- If this I²C path fails, the firmware may lose user input visibility even if the main safety loop still runs
